# TODO Generation (Exhaustive Checklists)

Generate formal, trackable TODO items with detailed acceptance criteria from planning artifacts. Every aspect of the project MUST be covered -- nothing is left implicit. TODOs should be **feature-sized** (not micro-tasks) so each one delivers a visible, verifiable change the user can see and interact with.

## Usage

```
/aether-omcc:todos
```

## Inputs

The skill reads planning artifacts in this priority order:

1. **Consensus plan** (required): `.omc/plans/ralplan-*.md` or `.omc/plans/consensus-*.md`
2. **Deep interview spec** (optional): `.omc/specs/deep-interview-*.md`
3. **Research artifacts** (optional): `.omc/research/` directory

If no plan is found, stop and instruct the user to run `/aether-omcc:ralplan` first.

## Outputs

- `.omc/todos/TODO-NNN.md` -- one file per task, following the format in `references/todo-format.md`
- `.omc/todos/INDEX.md` -- master index with summary tables and a dependency DAG

## Workflow

### Step 1: Read Planning Artifacts

1. Read the consensus plan from `.omc/plans/ralplan-*.md` or `.omc/plans/consensus-*.md`.
2. If `.omc/specs/deep-interview-*.md` exists, read it for additional context on requirements and constraints.
3. If `.omc/research/` contains files, read them for reference implementations and technical details.

### Step 2: Identify All Deliverables

Extract EVERY deliverable from the plan:
- Every data model and its fields/relationships
- Every API endpoint with methods, status codes, request/response shapes
- Every UI screen/page and its components/interactions
- Every integration point (third-party APIs, webhooks, events)
- Every auth flow (registration, login, logout, password reset, role checks)
- Every background job or scheduled task
- Every configuration or environment variable needed
- Every error handling requirement
- Every testing requirement (unit, integration, e2e)

### Step 3: Decompose into Feature-Sized Vertical Slices

Follow the decomposition rules in `references/decomposition-guide.md`:

- **Feature-sized, not micro-tasks**: Each TODO should deliver a complete, visible feature the user can see and verify. Bundle the data model, API endpoints, AND UI for a feature into ONE TODO. Example: "Implement user authentication (registration page, login page, JWT endpoints, user model)" is ONE TODO, not four.
- **Target 8-15 TODOs for a typical project**: A TODO list with 30+ items is too granular. Merge related deliverables into cohesive features. If you have more than 15 TODOs, consolidate — group by user-facing feature, not by technical layer.
- **Each TODO must produce a verifiable change**: After completing a TODO, the user should be able to open the app and see something different. A TODO that only creates a database migration with no API or UI is too small — bundle it with the feature that uses it.
- **Vertical slices**: Each TODO includes data + API + UI for its feature (mark sections N/A if not applicable). Never create separate TODOs for "create User model", "create User API", "create User page" — that's one TODO: "Implement user management".
- **Foundation TODOs are the exception**: Project setup, auth infrastructure, and base layout are allowed to be backend/config-only since they enable all subsequent features.
- **Domain-specific criteria**: Apply the checklist patterns from `references/checklist-patterns.md` based on task type.

### TODO Completeness Requirements

TODOs MUST describe COMPLETE features, not scaffolding:

**WRONG — Scaffolding TODO:**
- "Set up user model and basic auth routes"
- Acceptance: "Auth routes exist", "User model has fields"
- Result: Empty routes that return nothing, model with no data

**RIGHT — Complete TODO:**
- "Implement user authentication with registration, login, and session management"
- Acceptance: "User can register with email/password and see success message", "User can log in and is redirected to dashboard with their name shown", "Invalid login shows specific error message"
- Result: Working auth flow with real validation, real error messages, real redirects

Each TODO must specify:
- What DATA should exist (not just schema — actual seed data, realistic values)
- How components CONNECT (API returns data that UI displays, forms submit to endpoints that persist)
- What the user SEES at every step (not just "page renders" but "page shows 5 tasks with titles, dates, and status badges")

### Checklist Integration

If `.omc/checklists/project-checklist.md` exists (generated by `/aether-omcc:checklist`), read it and distribute relevant checklist items into each TODO's `## Playwright Verification` section. Each TODO should get the subset of checklist items that correspond to its feature. For TODOs with no UI changes, set the section to "No UI changes to test."

### Step 4: Write TODO Files

For each task, create `.omc/todos/TODO-NNN.md` using the template in `references/todo-format.md`.

Numbering rules:
- `TODO-001` through `TODO-NNN`, zero-padded to 3 digits.
- Foundation tasks get the lowest numbers.
- Number in dependency order: if TODO-005 depends on TODO-002, TODO-002 MUST have a lower number.

Each TODO MUST have:
- A clear, imperative title (e.g., "Implement user registration endpoint")
- A one-sentence summary explaining what it delivers and why
- A description covering Data, API, and UI aspects (mark N/A if not applicable)
- **At least 3 specific, testable acceptance criteria** as checkboxes
  - Criteria MUST contain concrete values (endpoint paths, status codes, field names, screen names)
  - Criteria MUST be independently verifiable
- A dependencies section listing prerequisite TODOs with reasons
- Priority: must-have (core MVP), should-have (secondary MVP), could-have (future/nice-to-have)
- Scope tags: `[frontend, backend, database, config]` as applicable

### Step 5: Generate INDEX.md

Create `.omc/todos/INDEX.md` containing:

1. **Summary**: Total TODO count, breakdown by priority and status.
2. **Table of all TODOs**: Columns: ID, Title, Priority, Status, Scope, Dependencies.
3. **Dependency DAG**: A text-based directed acyclic graph showing execution order. Use ASCII or Mermaid syntax. Group by execution tier (foundation -> core features -> secondary features -> polish).
4. **Critical path**: Identify the longest dependency chain and note it.

### Step 6: Coverage & Quality Verification

**This step is mandatory.** Before completing, verify ALL of the following:

**Coverage (every item in the plan is accounted for in at least one TODO):**
- [ ] Every API endpoint in the plan is covered by a TODO (multiple endpoints can be in one TODO)
- [ ] Every data model in the plan is covered by a TODO (bundled with the feature that uses it)
- [ ] Every UI screen/page in the plan is covered by a TODO (bundled with its backend)
- [ ] Every integration point in the plan is covered by a TODO
- [ ] Auth and authorization are fully covered
- [ ] Error handling, database setup, and configuration are covered

**Quality (TODOs are well-formed and actionable):**
- [ ] No TODO has fewer than 3 acceptance criteria
- [ ] All acceptance criteria contain concrete, testable values (endpoints, screen names, behaviors)
- [ ] Each TODO produces a visible, verifiable change (user can see/interact with the result)
- [ ] Dependencies form a valid DAG (no circular dependencies)
- [ ] Foundation tasks have no dependencies or only depend on other foundation tasks

**Size (not too many, not too few):**
- [ ] Total TODO count is between 5-15 for a typical project (adjust for project complexity)
- [ ] No TODO is so large it covers more than 3-4 screens or 8+ endpoints (split if so)
- [ ] No TODO is so small it only touches one layer without a visible result (merge if so)

If any check fails, merge or split TODOs as needed before proceeding.

## Completion

When all TODO files and INDEX.md are generated and the exhaustiveness verification passes, output:

```
PIPELINE_TODOS_COMPLETE
```
